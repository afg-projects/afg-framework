# 完善 AFG Framework PRD — 使用者视角全面评估与规范制定

## Goal

从使用者（特别是初学者）的角度，全面评估和完善 AFG Framework 的产品需求文档（PRD），使其成为一份"从零开始"即可快速上手、功能齐全、规范完整的框架需求文档。同时制定整个框架的开发规范体系。PRD 定位为产品需求蓝图，定义"框架应该达到什么状态"，引导框架向完整齐全的目标推进。

## What I already know

### 当前 PRD 状态（docs/framework-prd.md）
- 1389 行，覆盖 22 个模块、80 个 AutoConfiguration
- 架构描述扎实，但偏"实现清单"而非"使用指南"
- 缺少快速开始、模块选择指南、核心概念解释
- AI 模块几乎只是接口清单，无使用场景和示例
- 安全模块缺少配置步骤引导

### 代码库探查发现
- 实际 include 的子模块为 17 项（20 个模块），PRD 写的 22 个包含 ai-spring-ai
- ai-spring-ai 目录存在但**未被 settings.gradle.kts include**，属未完成模块
- core 模块确实有 31 个 AutoConfiguration
- **存在全局异常处理器** `GlobalExceptionHandler`（@RestControllerAdvice）
- **Bean Validation** 只引入了 `jakarta.validation-api`（API 层），未引入 `hibernate-validator` 运行时实现
- core 模块还有 `SchedulerAutoConfiguration`（PRD 中未列出）

## Decisions (confirmed)

1. **文档定位**：单一文档，产品需求蓝图，定义"框架应该达到什么状态"
2. **规范范围**：全量规范体系（8 大类：编码、API设计、异常处理、配置、实体设计、模块开发、依赖管理、安全）
3. **AI 定位**：核心模块，完整定义每个子功能
4. **未实现标注**：纯目标状态，不区分当前是否已实现
5. **文档结构**：分层递进式（概述 → 快速开始 → 核心概念 → 模块架构 → 逐模块需求 → 规范体系 → 配置参考 → 附录）
6. **描述模板**：6 段式（一句话定位 → 使用场景 → 期望 API 体验 → 完整功能清单 → 配置属性 → 注意事项/限制）
7. **扩展维度**：版本策略 + 脚手架规范 + 框架边界定义 + 方言完整度矩阵
8. **示例定位**：展示期望的 API 体验，可超越当前实现
9. **框架边界**：每个模块加"与 Spring Boot 原生详细对比"说明
10. **补齐策略**：全量补齐，利用 Spring Boot 增强，不重复造轮子
