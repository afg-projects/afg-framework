**评审范围**: afg-framework/core 模块

### 1.3 依赖关系

**下游模块对 Core 的依赖：**

| 模块 | 声明方式 | 实际使用 |
|------|----------|----------|
| data-core | api | 无 |
| security-core | api | BusinessException, PageData 等 4 个类 |
| ai-core | api | 无 |
| spring-boot-starter | api | 无 |

**关键发现：**
- `data-core` 和 `ai-core` 声明依赖 core 但无实际导入
- 只有 `security-core` 实际使用了 core 中的少量类（异常和结果模型）
- 真正使用 core 功能的是实现层模块（data-impl、security-impl）

## 二、拆分方案分析

### 2.1 历史拆分计划回顾

之前的拆分计划建议将 core 拆分为 6 个模块：
- core-base：异常、结果、工具类
- core-cache：缓存
- core-event：事件
- core-web：Web 增强
- core-module：模块系统
- core-observability：可观测性

### 2.2 拆分收益评估

| 收益项 | 预期收益 | 实际评估 |
|--------|----------|----------|
| 减少依赖传递 | 高 | **低** - 下游模块本就应依赖完整 core |
| 加快编译速度 | 高 | **极低** - 编译时间仅 0.9 秒 |
| 按需引入 | 中 | **低** - 大部分场景需要完整功能 |
| 依赖边界清晰 | 高 | **中** - 拆分后内部依赖更复杂 |

### 2.3 拆分成本评估

| 成本项 | 量级 |
|--------|------|
| 工程量 | 3-5 人天 |
| 迁移风险 | 中（API 变更、导入路径变更） |
| 维护成本 | 增加（多模块协调） |
| 文档更新 | 大量 |
| 下游适配 | 中（需要修改依赖声明） |

## 三、评审结论

### 3.1 结论：**不建议拆分**

### 3.2 理由

#### 3.2.1 编译时间不是问题

Core 模块编译时间仅 0.9 秒，拆分不会带来明显收益。

#### 3.2.2 依赖传递设计合理

当前设计中，下游模块通过 `api(project(":core"))` 获得 core 的全部能力，这是合理的：
- `data-core` 虽然无直接导入，但作为数据层抽象，拥有缓存、异常等能力是合理的
- `ai-core` 同理，作为 AI 层抽象，需要完整的基础能力

#### 3.2.3 拆分增加复杂度

如果拆分，会导致：
- 模块间依赖关系从 1 对 N 变成 N 对 M
- 版本协调成本增加
- API 兼容性测试工作量翻倍

#### 3.2.4 内部耦合度高

分析显示：
- `autoconfigure` 包依赖其他子包 120 处
- `web` 包依赖其他子包 52 处
- `scheduler` 包依赖其他子包 31 处

高耦合度意味着拆分边界难以划分，强行拆分会破坏封装。

### 3.3 替代建议

虽然不建议拆分，但可以做以下优化：

#### 3.3.1 清理无用依赖

```kotlin
// data-core 和 ai-core 可以移除对 core 的 api 依赖
// 如果确实需要异常类，可以仅依赖 commons 模块
```

#### 3.3.2 内部包结构优化

- 保持当前包结构
- 通过 `package-info.java` 明确各包职责
- 文档化各子包的使用场景

#### 3.3.3 考虑 `commons` 模块增强

将真正通用的类（如 BusinessException、PageData）下沉到 `commons` 模块，让不需要完整 core 功能的模块只依赖 `commons`。

## 四、最终建议

| 行动 | 优先级 | 预估工作量 |
|------|--------|-----------|
| 清理 data-core/ai-core 对 core 的无用依赖 | P2 | 0.5 人天 |
| 将 BusinessException 等通用类移至 commons | P3 | 1 人天 |
| 保持 core 模块不拆分 | P0 | 0 |
